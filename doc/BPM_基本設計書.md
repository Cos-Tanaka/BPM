## 1. 目的・背景

### 1.1 背景

現在、エンハンス案件はBacklog上で管理されているが、以下の課題があり案件全体の状況把握が困難な状態となっている。

- **課題の多層化：** 1案件あたり複数の子課題（工程別）が存在し、Backlogの標準一覧では案件単位の進捗が把握しづらい
- **進捗の分散：** 詳細設計・単体テスト設計・製造・単体テスト実施それぞれの状況を個別に確認する必要があり、全体像の把握に手間がかかる
- **遅延リスクの見落とし：** 案件横断での進捗遅れリスクを自動・定量的に検知する手段がない

### 1.2 目的

本システム（BPM）は、以下の目的を達成するために構築する。

1. **案件単位での進捗一覧表示：** 親課題（案件）を軸に、全子課題の進捗を集約して一覧表示する
2. **進捗遅れリスクの自動検知：** 期限・完了状況から進捗遅延リスクを自動判定し、アラートとして提示する
3. **即時性の確保：** 内部DBを持たず、常にBacklog APIから最新情報を取得して表示する

### 1.3 対象ユーザー

- エンハンス担当PM・リーダー
- 進捗管理を行う管理者

## 2. システム概要

### 2.1 システム全体像

```
[ブラウザ]
    |
    | HTTP
    v
[BPM Webサーバ（Node.js / Python等）]
    |
    | HTTPS (Backlog API)
    v
[Backlog クラウドサービス]
```

- ユーザーはブラウザ経由でBPM Webサーバにアクセスする
- BPM Webサーバは、ユーザーのリクエストに応じてBacklog APIを呼び出し、情報を取得・加工して返却する
- 取得した情報は毎回APIから取得し、内部DBへのデータ保存は行わない

### 2.2 主な機能

| 機能 | 概要 |
|---|---|
| 案件一覧表示 | 親課題（案件）単位で子課題の進捗を集約して一覧表示 |
| 進捗ステータス表示 | 工程（詳細設計／単テ設計／製造／単テ実施）別の完了状況を表示 |
| 進捗遅れリスク検知 | 工数ベースの2軸判定で遅延リスクを算出し、視覚的に警告表示 |
| Backlog直接リンク | 案件キーをクリックするとBacklogの当該課題画面を別タブで開く |

## 3. 用語定義

| 用語 | 定義 |
|---|---|
| 案件 | Backlog上の親課題。種別 `00.案件` 、かつカスタム項目「ステータス」が40〜60のもの |
| 子課題 | 案件（親課題）に紐づく工程別の課題 |
| 工程 | 詳細設計 / 単体テスト設計 / 製造 / 単体テスト実施 の4種 |
| 予定工数 | 親課題 + 子課題の `estimatedHours` 合計 |
| 実績工数 | 親課題 + 子課題の `actualHours` 合計 |
| 残工数 | 進捗率・実績工数をもとに算出した未完了工数 |
| テストリリース予定日 | 親課題カスタム項目「テストリリース予定日」の値 |
| 営業日残日数 | テストリリース予定日までの土日・非営業日を除いた日数 |
| 遅延リスク | 納期余裕率と進捗遅延率の2軸で算出した遅れ危険度 |
| BPM | Backlog Progress Monitor の略称 |

## 5. Backlogデータ構造

### 5.1 課題階層

```
案件（親課題）
└── 子課題①：詳細設計（機能A）
└── 子課題②：詳細設計（機能B）
└── 子課題③：単体テスト設計（機能A）
└── 子課題④：製造（機能A）
└── 子課題⑤：単体テスト実施（機能A）
└── ...
```

### 5.2 抽出条件（親案件）

BPMが一覧表示する「親案件」は、以下の条件をすべて満たす課題に限定する。

| # | 条件 | 値 |
|---|---|---|
| 1 | プロジェクト | `V2A9`（設定ファイルで変更可能）|
| 2 | 課題種別 | `00.案件` |
| 3 | Backlogステータス | `未対応` または `処理中` |
| 4 | 親子関係 | 親課題のみ（`parentIssueId` が `null`）|
| 5 | カスタム項目「ステータス」 | 先頭数値が `40` 以上 `60` 以下（例: `40【C】開発-指示有` 〜 `60【C】本番リリース-依頼有`）|

> ※ 条件5により、見積・要件定義フェーズ（30番台）や完了済み（70番台以降）は対象外となる

### 5.3 使用するBacklog課題フィールド

#### 標準フィールド

| フィールド | 用途 |
|---|---|
| `id` | 課題ID |
| `issueKey` | 課題キー（例：V2A9-123）|
| `summary` | 課題名 |
| `status` | ステータス（未対応／処理中／処理済み／完了）|
| `issueType` | 課題種別（`00.案件` での絞り込みに使用）|
| `parentIssueId` | 親課題ID（子課題取得・抽出条件判定に使用）|
| `assignee` | 担当者 |
| `milestone` | マイルストーン |
| `startDate` | 開始日（営業日計算の起点。無ければ `created` を使用）|
| `dueDate` | 期限日 |
| `estimatedHours` | 予定工数（h）|
| `actualHours` | 実績工数（h）|

#### カスタムフィールド

| カスタム項目名 | 用途 |
|---|---|
| `ステータス` | 抽出条件（40〜60番台のみ対象）|
| `テストリリース予定日` | 営業日残日数計算の基準日 |
| `進捗率` | 残工数計算の優先入力値 |

## 6. 機能要件

### 6.1 案件一覧表示

- 5.2節の抽出条件を満たす親課題（案件）を一覧表示する
- 各案件行に以下の情報を表示する

| 表示項目 | 内容 |
|---|---|
| 案件キー | Backlogの課題キー。クリックすると当該Backlog課題ページを**別タブ**で開く |
| 案件名 | 親課題のsummary |
| 担当者 | 親課題のassignee |
| テストリリース<br>予定日 | カスタム項目「テストリリース予定日」の値（`YYYY/MM/DD`形式）。下段に「残営業日数」バッジを表示。※「本番リリース待ち」時はバッジのみ表示。 |
| 工数 (予/実/残) | 親+子の `estimatedHours` / `actualHours` / 算出した `残工数` を縦並びリストで表示 |
| 工程別進捗 | 各工程の完了数／全数（詳細設計・単テ設計・製造・単テ実施）|
| 遅延リスク | リスクレベル（🟢正常 / 🟡注意 / 🔴危険 / ⚠️要確認）|

### 6.2 進捗遅れリスク検知

- 遅延リスクを自動計算し、一覧上で視覚的に警告表示する
- 詳細は [9章](#9-進捗遅れ検知ロジック) を参照

### 6.3 フィルタ・検索

- 担当者フィルタ（取得した案件データに存在する担当者名から動的にリストを生成）
- リスクレベルフィルタ（選択したレベルごとに表示を即時切り替え）

## 8. Backlog API連携

### 8.1 使用APIエンドポイント（予定）

| 処理 | エンドポイント | メソッド |
|---|---|---|
| 課題一覧取得（親課題） | `GET /api/v2/issues` | GET |
| 子課題一覧取得 | `GET /api/v2/issues?parentIssueId[]={id}` | GET |
| プロジェクト情報取得 | `GET /api/v2/projects/{projectIdOrKey}` | GET |

### 8.2 認証方式

- Backlog APIキーを使用（`apiKey` クエリパラメータ）
- APIキーは環境変数（`.env`）で管理し、フロントエンドには直接渡さない

### 8.3 データ取得フロー

```
ブラウザ → BPMサーバ（リクエスト）
BPMサーバ → Backlog API（親課題一覧取得）
BPMサーバ → Backlog API（各親課題の子課題取得）
BPMサーバ → 集計処理（工程分類・進捗率・遅延リスク算出）
BPMサーバ → ブラウザ（集計済みデータ返却）
```

### 8.4 環境変数定義（予定）

| 変数名 | 内容 | 例 |
|---|---|---|
| `BACKLOG_API_KEY` | Backlog APIキー | `xxxxxxxxxxxxxxxx` |
| `BACKLOG_SPACE` | Backlogスペース名 | `myspace` |
| `BACKLOG_PROJECT_KEY` | 対象プロジェクトキー | `V2A9` |
| `PORT` | サーバ起動ポート | `3000` |
| `HOLIDAY_CSV_PATH` | 非営業日一覧CSVのパス（UNCパス可）| `\\\\SERVER\\Document\\...\\非営業日一覧.csv` |

## 9. 進捗遅れ検知ロジック

### 9.1 基本方針

- **工数ベース**で残工数を算出し、「納期余裕率」と「進捗遅延率」の**2軸**で遅延リスクを判定する
- テストリリース予定日までの**営業日残日数**を基準に利用可能工数を計算する
- 判定に必要なデータが揃わない場合は `⚠️ 要確認` として扱う

### 9.2 残工数の算出

親課題1件に対して、親課題自身 + 子課題の値を合算して算出する。

#### 9.2.1 予定工数・実績工数

- **予定工数** = 親課題の `estimatedHours` + 子課題の `estimatedHours` 合計
- **実績工数** = 親課題の `actualHours` + 子課題の `actualHours` 合計

#### 9.2.2 残工数の優先順位

各課題（親/子）ごとに以下の優先順位で残工数を算出し、親案件単位で合算する。

| 優先順位 | 条件 | 計算式 |
|---|---|---|
| 1位 | カスタム項目「進捗率」が設定されている | `残工数 = 予定工数 × (1 - 進捗率 / 100)` |
| 2位 | 進捗率なし、実績工数あり | `残工数 = 予定工数 - 実績工数` |
| 3位（フォールバック） | 進捗率・実績工数ともになし | `残工数 = 予定工数` |

> ※ 予定工数が未設定の課題は残工数を計算しない

### 9.3 テストリリース予定日の取得

- 親課題のカスタム項目 `テストリリース予定日` を使用する
- 受け入れ形式: `YYYY-MM-DD` / `YYYY-MM-DDTHH:MM:SSZ` / タイムゾーン付き形式

### 9.4 営業日計算

営業日の判定には以下のルールを適用する。

| ルール | 内容 |
|---|---|
| 週末除外 | 土曜・日曜を非営業日として除外 |
| 非営業日CSV | 環境変数 `HOLIDAY_CSV_PATH` で指定したCSVファイルの日付を追加除外 |
| CSV読み込み失敗時 | 警告ログを出力し、「土日除外のみ」で計算を継続 |
| 1営業日あたりの稼働時間 | 8h |
| 開始日 | `startDate` を使用。未設定の場合は課題の `created`（作成日）を使用 |

### 9.5 遅延リスク判定（2軸）

#### 9.5.1 本番リリース待ち（例外処理）

カスタム項目「ステータス」が `59` （例: `59【Ａ】テストリリース - 済`）から始まる場合、実質的に完了しているとみなし、以下の特別な扱いとする。

- リスク判定は無条件で `🟢 正常` とする。
- 画面上のテストリリース予定日と残日数の表示を行わず、「本番リリース待ち」のバッジのみを表示する。

#### 9.5.2 要確認（データ不備）

以下いずれかの条件を満たす場合は `⚠️ 要確認` とし、2軸判定を行わない。（※例外として、すべて完了または本番リリース待ちの場合は正常として扱う）

- テストリリース予定日が未設定
- 残工数が未算出
- 予定工数が未設定または0以下

#### 9.5.3 軸1：納期余裕率

| 変数 | 定義 |
|---|---|
| `A` | 利用可能工数 = 営業日残日数 × 8(h) |
| `M` | 納期余裕率 = `(A - 残工数) / A` |

| 判定 | 条件 |
|---|---|
| 🔴 危険 | `A <= 0` または `M < 0.15` |
| 🟡 注意 | `0.15 <= M < 0.25` |
| 🟢 正常候補 | `M >= 0.25` |

#### 9.5.3 軸2：進捗遅延率

| 変数 | 定義 |
|---|---|
| `E` | 経過比率 = 経過営業日 / 全営業日（開始日〜テストリリース予定日）|
| `P` | 実績進捗比率 = `1 - (残工数 / 予定工数)` |
| `L` | 遅延率 = `E - P` |

| 判定 | 条件 |
|---|---|
| 🔴 危険 | `L > 0.15` |
| 🟡 注意 | `0.08 < L <= 0.15` |
| 🟢 正常候補 | `L <= 0.08` |

#### 9.5.4 最終リスクレベル決定

| 最終判定 | 条件 |
|---|---|
| 🔴 危険 | いずれかの軸が「危険」 |
| 🟡 注意 | 危険なし、いずれかの軸が「注意」 |
| 🟢 正常 | 両軸とも「正常候補」 |

> ※ 閾値（0.15 / 0.25 / 0.08 / 0.15）は運用後にチューニングする

## 10. 環境構成

### 10.1 本番環境

| 項目 | 内容 |

## 11. 非機能要件

| 項目 | 要件 |
|---|---|
| パフォーマンス | 案件一覧の表示は10秒以内（Backlog API応答時間依存）|
| セキュリティ | APIキーはサーバサイドのみで管理、フロントエンドに露出させない |
| 可用性 | 本番環境はDockerのリスタートポリシーで自動再起動 |
| メンテナンス性 | 環境変数で設定変更可能（コード変更不要）|
| ブラウザ対応 | Chrome最新版 / Edge最新版 |

## 12. 制約・前提条件

- Backlog APIの呼び出しレート制限（600リクエスト/分）の範囲内で設計する
- 対象プロジェクトは **`V2A9`（1プロジェクトのみ）** に固定する
- 内部DBは持たず、表示のたびにAPIを呼び出すため、大量案件存在時はパフォーマンスに注意が必要
- 案件一覧はページングなしのスクロール表示とする（全件を一度に表示）
- 認証・アクセス制御は実装しない（社内イントラネット内での利用を前提）
- 通知機能（メール・Slack等）は実装しない
- 本番環境のLinuxサーバおよびDocker環境は別途用意する（本設計書のスコープ外）
